# Data Model and Tests

This page covers the persistent data model, the normalization pipeline, and the tests that provide the most value when changing scraping or persistence behavior.

## Persistent data model

The SQLite backend stores three related record types in `data/scraper.db`:

### `searches`

A single row per search request.
It stores:

- search id
- query
- provider
- URL
- status
- product count
- execution time
- raw provider JSON snapshot
- pagination metadata
- schema version
- timestamp

### `raw_products`

A row per raw product returned by a provider.
It stores:

- provider product identity if available
- title
- price text
- parsed numeric price
- product URL
- serialized raw JSON

### `normalized_products`

A row per normalized product generated by `src/providers/normalizer.js`.
It stores:

- original and normalized titles
- brand and model
- storage and memory capacity
- normalized currency and pricing fields
- availability
- confidence score
- the original price text

The repository code is already structured for future additions such as canonical IDs and price history.

## Normalization rules

The normalizer is the shared cross-provider glue.
It:

- strips decorative or promotional noise from titles
- detects common hardware brands
- extracts product models using regular expressions and fallback heuristics
- parses Brazilian price formats
- infers availability using title, price text, and provider-specific behavior

The detailed rule set is documented in `docs/NORMALIZATION.md`, but the source of truth is `src/providers/normalizer.js`.

## Best tests to run after changes

### `tests/normalization.test.js`

Use this when changing title cleanup, brand extraction, price parsing, or availability logic.
It provides the strongest signal for whether the shared normalizer still matches expected behavior.

### `tests/provider-regression.test.js`

Use this when changing provider selectors, pagination, or browser timing behavior.
It exercises Pichau, KaBuM, and Mercado Livre with a representative set of search queries.

### Historical smoke tests

Files named `tests/test-*.js` are older focused provider checks.
They are useful as supporting evidence, but the regression test and the normalization unit test are the more important checks to preserve.

## Failure behavior to remember

`SearchService` records failed searches as well as successful ones.
That means a provider failure does not simply return an API error; it also leaves a database record that downstream analysis can inspect.

## Change checklist

Before finishing a provider or normalization change:

1. Confirm the API still returns raw products and normalized products.
2. Confirm search metadata is persisted even on failure.
3. Re-run normalization tests.
4. Re-run provider regression tests.
5. Check that database records still include the expected fields.
